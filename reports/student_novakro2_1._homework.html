Řešení problému batohu metodou hrubé síly a jednoduchou heuristikou

Specifikace úlohy

Problém batohu je jeden z nejjednodušších a nejznámějších NP-problémů. Je dáno n (počet věcí), M (maximální kapacita batohu), konečná množina dvojic (c, v), která reprezentuje možné věci do batohu, kde c je cena a v je váha. A chceme najít takovou kombinaci věcí, které mají maximální hodnotu a vejdou se do batohu (tedy součet hmotností =< M).

Možné varianty řešení

Tento problém se dá řešit:

  • hrubou silou - vyzkoušením všech kombinací
  • jednoduchou heuristikou - pomocí greedy heuristiky (klesající ceny, poměru cena/váha)
  • dynamickým programováním
  • metodou větvení hranic
  • FPTAS algoritmu

Postup řešení

Problém jsem řešil (dle zadání) hrubou silou a pak jednoduchou heuristikou pomocí klesajícího poměru cena/váha. Pro měření výsledku jsem si do programů také napsal automatický vstup instancí a výstup do formátu csv se všemi potřebnými informacemi. Takže výsledný program (řešení) byl následující:

  1. Načti soubor.
  2. Načti řádek souboru a vytvoř instanci.
  3. Najdi řešení algoritmem hrubé síly a změř jeho čas.
  4. Najdi řešení pomocí jednoduché heuristiky a změř jeho čas.
  5. Porovnej výsledky a spočítej relativní chybu.
  6. Zapiš do souboru výsledky měření, relativní chybu, čas
  7. Opakuj od 2. dokud jsou ještě řádky v souboru.
  8. Zapiš do souboru další informace o měření. (Průměrný čas, …)
  9. Opakuj od 1. dokud jsou ještě soubory.

Naměřené výsledky jsem pak následně vhodným způsobem zpracoval/upravil a vytvořil z nich grafy a také tabulku výsledků.

Popis kostry algoritmu

Řešení hrubou silou

  function solveKnapsackProblem(List<Pair<int,int>> things,int M)
      ...
      maxWeight = ... // Passed by argument
      things = ... // Passed by argument
      maxValue = 0
      solutionCombination = 0
      ...
      for combination = 0 ... 2^n
          valueAndWeight = countValueAndWeight(things, combination)
          if(valueAndWeight.value > maxValue AND valueAndWeight.weight =< maxWeight)
              maxValue = valueAndWeight.value
              solutionCombination = combination
          end
      end
      ...
  end
  function countValueAndWeight (List<Pair<int,int>> things, int combination)
      value = 0
      weight = 0
      for i = 0...things.length
          if combination & 1
              value += things[i].value
              weight += things[i].weight
          end
      end
      return Pair(value,weight)
  end

Řešení jednoduchou heuristikou

  function solveKnapsackProblem(List<Pair<int,int>> things,int M)
      ...
      maxWeight = ... // Passed by argument
      things = ... // Passed by argument
      value = 0
      ...
      for i = 0 ... things.length
          ratios[i] = Pair(things[i].value / things[i].weight, i)
      end
      ratios = sort(ratios) // a > b
      for i = 0 ... things.length
          if things[ratios[i].index].weight + weight > maxWeight
              break
          end
          value += things[ratios[i].index].value
          weight += things[ratios[i].index].weight
      end
      ...
  end

Závislost času na velikosti instance

Graf neobsahuje čas řešení pomocí heuristiky, protože ten byl o hodně řádu rychlejší než pomocí hrubé síly. Ale čas nebyl ani milisekunda. Na grafu je vidět, že čas se exponencielně zvyšoval na závislosti velikosti instance, což odpovídá počtu možnosti, které musí projít algoritmus, což je 2^n.

Čas byl měřen na platformě Linux Ubuntu 15.04 a program byl napsán v jazyce C++11.

Závěr

Podařilo se mi naimplementovat exaktní řešení a řešení pomocí jednoduché heuristiky NP-problému batohu. Dle naměřených výsledků je vidět, že:

  • u exaktního řešení čas řešení exponenciálně roste s rostoucím n
  • u řešení jednoduchou heuristikou nemá skoro vliv n na čas řešení oproti řešení hrubou silou
  • řešení pomocí jednoduché heuristiky nemusí být řešení nejlepší
  • relativní chyba se pohybovala nejčastěji kolem 1%, ale maximální chyba se pohybovala kolem 5%

Takže pokud nehledáme nejlepší řešení pro vysoké n, tak můžeme použít jednoduchou heuristiku a s chybovostí 1%. Ale pokud bychom chtěli co nejlepší řešení hodilo by se uvažovat o použití algoritmu hrubou silou a ten paralelizovat na SIMD počítače, tím bychom dostali řešení mnohem rychleji pro velké n.
